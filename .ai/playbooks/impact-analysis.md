# Playbook — Impact Analysis (영향도 분석)

> AI는 Playbook을 가장 잘 따른다. 각 단계를 순서대로 수행하고, 근거를 남긴다.

## 목적
이슈/변경이 코드·API·DB·운영에 미치는 영향을 산출하고 보고서로 만든다.

## 입력
- `issue_key` (예: `PROJ-123`) 또는 변경(diff/커밋 범위)

## 절차
### 1. Jira 조회
- 이슈 본문/타입/상태/우선순위/연결 이슈 수집.
- 산출: 이슈 요약, 관련 링크.

### 2. 댓글 조회
- 이슈 코멘트에서 결정/재현/영향 언급 추출.
- 산출: 맥락 노트(무엇이 논의됐나).

### 3. Git 조회
- 이슈에 연결된 커밋/PR, 변경된 파일 목록.
- 산출: 변경 파일 집합, 최근 관련 커밋.

### 4. DB 영향도
- 변경 파일이 건드리는 테이블/모델, 마이그레이션 여부.
- 그래프(Neo4j)로 `File-[:DEPENDS_ON]->File` 확장.
- 산출: 영향받는 데이터/모듈 목록.

### 5. API 영향도
- 변경이 노출하는 엔드포인트/계약 변화(호환성 파괴 여부).
- 산출: 영향 API 목록 + 호환성 판정.

### 6. 보고서 작성
- 요약 → 영향 범위(코드/API/DB) → 리스크 → 권장 조치.
- 산출: `impact_reports` 저장 + 리포트 응답.

## 데이터 소스 규칙
모든 외부 조회는 `infrastructure/{jira,git,postgres,neo4j}` 를 경유한다. 프롬프트는 `prompts/impact/`.

## 산출물 형식(요약)
```
[요약] 한 문단
[영향-코드] 파일/모듈 리스트
[영향-API]  엔드포인트 + 호환성
[영향-DB]   테이블 + 마이그레이션 필요?
[리스크]    상/중/하 + 근거
[권장]      해야 할 조치
```
