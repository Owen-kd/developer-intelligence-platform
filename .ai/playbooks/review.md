# Playbook — Code Review (코드 리뷰 보조)

> 변경(diff/PR)에 대한 리뷰 초안을 만든다. 최종 판단은 사람이 한다.

## 입력
- diff / PR / 커밋 범위 (+ 관련 이슈)

## 절차
### 1. 변경 요약
- 무엇이/왜 바뀌었는지 한 문단. 관련 이슈 맥락 연결.

### 2. 아키텍처 점검
- 의존성 방향, 모듈 경계, 외부호출 격리(infrastructure) 위반 여부.

### 3. 정확성 점검
- 엣지케이스/에러 경로, 동시성, LLM 출력 검증 유무.

### 4. 품질 점검
- 네이밍/스타일/타입/테스트 충분성/중복·복잡성.

### 5. 리스크 판정
- Blocking / Non-blocking / Nit 로 분류, 각 항목에 파일:라인 근거.

### 6. 리뷰 노트 작성
- 템플릿: [../templates/review.md] → [../reviews/].
- 프롬프트: `prompts/review/`.

## 규칙
- 지적에는 항상 구체적 위치와 근거. 추측성 지적 금지.
- 개선안은 "왜"와 함께. 스타일 취향은 Nit으로 낮춘다.
- 이 플레이북은 [../workflow/04-review.md] 게이트와 정렬된다.
