# Coding Guidelines

## 언어 · 스타일
- Python 3.11+. **타입 힌트 필수** (public 함수/메서드/클래스 속성).
- 포매팅/린트: `ruff` (line-length 100). 커밋 전 `ruff check` / `ruff format` 통과.
- 타입 체크: `mypy --strict`. 새 코드는 strict 통과를 지향한다.
- 비동기 우선: I/O(DB, HTTP, LLM 호출)는 `async`. 블로킹 라이브러리는 executor로 감싼다.

## 함수 · 클래스
- 함수는 한 가지 일만 한다. 20~30줄을 넘으면 분리를 고려한다.
- 부수효과(외부 호출, 쓰기)는 이름과 위치로 드러낸다. 순수 로직과 섞지 않는다.
- 매직 넘버·문자열은 `shared/constants/` 로 뺀다.

## 에러 처리
- 도메인 에러는 `shared/exceptions/` 의 커스텀 예외로 표현한다.
- 예외를 조용히 삼키지 않는다. 단, 헬스체크처럼 "실패해도 진행"이 의도면 주석으로 명시한다.
- 사용자에게 나가는 메시지와 내부 로그 메시지를 구분한다.

## 로깅
- `print` 금지. `shared/logger/` 를 사용한다.
- 구조화 로깅(키=값). 요청/이벤트 추적을 위한 correlation id를 남길 수 있게 한다.

## 테스트
- 새 로직에는 테스트를 붙인다. 위치는 `tests/{unit,integration,e2e}/`.
- 외부 시스템(LLM, Jira, DB)은 unit 테스트에서 목(mock)한다. 실제 연동은 integration에서.
- 테스트는 결정적이어야 한다. 시간·랜덤·네트워크 의존을 격리한다.

## 커밋
- 작고 논리적인 단위로 커밋한다. 한 커밋에 포맷팅 + 로직 변경을 섞지 않는다.
- 메시지는 "무엇을/왜"를 담는다.

## LLM 코드 작성 시
- 프롬프트 문자열을 코드에 인라인하지 않는다 → `prompts/` 파일 참조.
- LLM 응답은 항상 검증한다(스키마/파싱 실패 대비). 모델 출력을 신뢰해서 바로 실행하지 않는다.
