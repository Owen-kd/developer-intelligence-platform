# Sprint-15 — 지식 어시스트 (Inquiry Assist + Legacy Rationale)

- 상태: Discovery/Design 완료, 구현 대기
- Phase: 실사용 가치 (DIP의 핵심 목적 실현)
- 관련: [ADR-006](../decisions/ADR-006-knowledge-retrieval-architecture.md) · [Sprint-14](Sprint-14.md)(실 어댑터)

## 문제 (Discovery) — 사용자 실제 페인포인트
1. **1:1 문의 처리 지연**: 문의가 Jira 이슈로 등록되면 대부분 **기획/개발(툴팀·엔진팀) 확인이 필요한 크리티컬 건**.
   개발자 확인/조사에 **5~6시간** 소요(매번 "무슨 문제? 전에 있었나? 원인? 어디 코드?"를 맨땅에서).
2. **레거시 "왜" 실종**: 기획한 **기획자들이 퇴사** → 레거시가 "왜 이렇게 됐는지"를 아무도 모름 →
   개발자가 매번 역추적하느라 오래 걸림(= 회사의 기억이 사람 머릿속에만 있다 증발).

## 사용자 확인 사실 (인터뷰)
- Jira 이슈 = 기획/개발팀 확인 대상의 크리티컬 건.
- **팀 구분: 프로젝트 키 접두사** — `ENG-` = 엔진팀, `PA20-` = 툴팀 (라우팅 신호로 사용).
- **해결 방법이 코멘트에 잘 문서화됨** → 지식 자산화 가능(결정적 전제 충족).
- **전달 채널: Jira 코멘트 자동 + Slack + NaverWorks** (팀이 일하는 곳으로 push).

## 두 use case = 같은 엔진
| use case | retrieval | LLM 종합 |
|----------|-----------|----------|
| ① 새 문의 처리 보조 | 유사 **해결된** 과거 이슈 | 원인 + 조치 + 담당 |
| ② 레거시 "왜" 재구성 | 관련 **기획/결정** 이슈 + **구현 커밋(링크)** | 배경·의도·제약·변경 이력 |

둘 다 **`과거 Jira+Git 검색 → LLM이 근거와 함께 조립 → 팀 채널로 전달`**.

## 아키텍처 방향 (ADR-006)
순수 벡터 RAG가 아니라 **하이브리드**:
Postgres(원천) + **Knowledge 증류** + 하이브리드 검색(키워드+벡터) + **그래프(이슈↔커밋)** + **근거 인용 강제**.
- "왜" 재구성은 유사도가 아니라 관계·시간축 문제 → 그래프/구조가 필수(벡터만으론 약함).
- 벡터는 스케일 시 **pgvector**로 도입, 임베딩은 **로컬 다국어 모델(프라이버시)**. 상세: ADR-006.

## MVP 정의 (지금 있는 것으로 구현 가능)
`assist <이슈키 또는 텍스트>` → 하나의 브리핑:
- 관련 과거 이슈 top-N + 왜 관련되는지
- 배경·의도 (기획 이슈/코멘트 기반)
- 추정 원인 / 과거 해결책
- **담당 추천** (프로젝트 접두사: ENG→엔진팀, PA20→툴팀)
- **확신도 + 근거 링크** (이슈키, 커밋 sha) — 인용 없으면 "확실치 않음"
- retrieval = Jira 자체 검색(한국어 색인), synth = LLM(이미 연동됨). 벡터 없이 시작.

## 안전장치 (절대)
- **근거(이슈키/커밋sha) 필수 인용** → 환각 방지(퇴사자 지식 복원의 최대 리스크).
- **사람 확인 후 발송**: AI는 제안, 운영/개발이 확인 후 전달. 고객 자동응답 X.
- **피드백 루프**: 해결될 때마다 다시 Knowledge로 축적 → 쓸수록 똑똑해짐.

## 구현 단계
1. `assist` 엔진: Jira 검색 retrieval + LLM 종합 + 근거 인용 + 팀 라우팅(접두사).
2. 전달 어댑터: `infrastructure/jira`(코멘트 write) → `infrastructure/slack` → `infrastructure/naverworks`. fan-out.
3. 스케일: 백필(과거 해결 이슈 수천 건) + Knowledge 증류(해결책 추출) + pgvector(로컬 임베딩).
4. 트리거: 스케줄러가 신규 이슈 감지 → assist 실행 → 채널 전달.
