# ADR-012 — LLM 게이트웨이 기반 중앙 거버넌스 (공급망 격리 설계)

- 상태: **Proposed** (승인 대기 — APR 게이트 필요, 아래 참조)
- 날짜: 2026-07-10
- 관련: [ADR-006](ADR-006-anthropic-adapter.md)(Anthropic 어댑터) · [ADR-004(제안)](../architecture/knowledge-lifecycle.md)(AI는 Knowledge만 소비) · [ADR-010](ADR-010-team-shelf-access-control.md)(접근제어) · [APR-005](../planning/approvals/APR-005-llm-vendor-data.md)

---

## 한눈에 (TL;DR)

지금은 각 코드가 Anthropic/OpenAI를 **제각각** 부른다. 이걸 **문 하나(게이트웨이)** 로 모아
키 관리·예산·감사·정책을 한 곳에서 통제하려 한다.

문제: 문 하나에 모으면 **키가 한곳에 몰려서**, 그 문(게이트웨이)이 쓰는 라이브러리에
악성코드가 섞이면 `.env`(키·크레덴셜)가 통째로 샐 수 있다.

해결: 문을 만들되, 그 문을 **작은 독방에 가두고(격리)**, **밖으로 나가는 출구를 LLM 회사 주소로만
열어둔다(egress 제한).** 그러면 악성코드가 키를 훔쳐도 **바깥으로 내보낼 길이 없다.**

---

## 맥락 (왜 이걸 지금 고민하나)

### 게이트웨이가 뭔가 — 쉽게
"LLM 게이트웨이"는 우리 코드와 LLM 회사(Anthropic/OpenAI) 사이에 서는 **중간 창구**다.
지금은 이렇다:

```
[wiki_service] ─┐
[triage agent] ─┼─▶ 각자 직접 ─▶ Anthropic / OpenAI
[ask API]      ─┘   (키도 각자, 로그도 각자, 예산 통제 없음)
```

게이트웨이를 두면 이렇게 바뀐다:

```
[wiki_service] ─┐
[triage agent] ─┼─▶ [게이트웨이] ─▶ Anthropic / OpenAI
[ask API]      ─┘    ↑ 여기서 한 번에:
                       - 키 관리 (한 곳)
                       - 예산/횟수 제한
                       - 감사 로그 (누가 뭘 물었나)
                       - 모델 허용목록
                       - 실패 시 다른 모델로 우회
```

**얻는 것**: 통제가 한 곳에 모인다(거버넌스). 키를 코드마다 뿌릴 필요 없다.

### 그런데 걱정되는 지점 (오너 문제 제기)
문을 하나로 모으면 그 문 뒤에 **모든 키가 쌓인다.** 게이트웨이는 보통 의존성(라이브러리)이 많은데,
그중 하나에 악성코드가 섞이면(요즘 PyPI/npm에서 실제 일어나는 **공급망 공격**) —
그 프로세스가 볼 수 있는 `.env`를 읽어서 **외부 서버로 몰래 전송**할 수 있다.

즉 게이트웨이는 **거버넌스 이득 = 위험 집중** 이라는 양날의 칼이다.
이 ADR의 핵심은 "이득은 취하되 위험 집중을 어떻게 격리하나"이다.

## 결정

게이트웨이를 도입하되, **두 가지를 분리**하고 **네 겹으로 격리**한다.

### 1) 두 평면을 분리한다 (개념)

| 평면 | 뜻 | 어디서 담당 |
|------|------|------------|
| **운영 거버넌스** | 키·예산·감사·라우팅 (기계적 통제) | 게이트웨이 OK |
| **데이터 거버넌스** | "raw 데이터 LLM 직송 금지", "Knowledge만 소비" | **우리 코드**(Context Builder)에 유지 |

> 이유: 헌법의 `Context Before AI` / [ADR-004(제안)] 규칙은 **서드파티에 맡기면 안 된다.**
> 게이트웨이는 "무엇을 보낼지"가 아니라 "어떻게/누구에게 보낼지"만 통제한다.
> 무엇을 보낼지(정제·맥락화)는 게이트웨이 **호출 전에** 우리 코드가 이미 끝낸다.

### 2) 네 겹으로 격리한다 (공급망 → .env 탈취 직접 방어)

핵심 비유: **"금고(키)를 독방에 넣고, 그 방의 유일한 문은 은행으로만 통하게 한다."**

1. **독방에 가둔다 — 프로세스/컨테이너 격리 + 최소 키만**
   게이트웨이는 별도 컨테이너로 띄우고, **LLM provider 키만** 넣는다.
   DIP의 Postgres·Jira·Redis 크레덴셜은 **그 방에 아예 두지 않는다.**
   → 악성코드가 그 방의 `.env`를 다 읽어도, 훔칠 게 provider 키뿐(나머지는 애초에 없음).

2. **출구를 막는다 — Egress(외부로 나가는 통신) 허용목록** ⭐ 가장 확실한 방어
   게이트웨이 컨테이너가 밖으로 연결할 수 있는 주소를
   `api.anthropic.com` / `api.openai.com` **딱 두 곳으로만** 제한한다.
   → 악성코드가 키를 훔치는 데 성공해도, `attacker.com`으로 **보낼 길이 없다.**
   "훔치기"는 못 막아도 "빼돌리기"를 막으면 공격이 완성되지 않는다.

3. **금고를 파일로 두지 않는다 — 런타임 주입**
   키를 디스크의 `.env` 파일로 두면 `파일 열기` 한 번에 털린다.
   secrets manager(또는 최소한 짧은 수명 토큰)로 **실행 시점에 주입**한다.
   → "파일 읽기" 공격의 수확이 없어진다.

4. **재료를 검사한다 — 의존성 고정 + 스캔**
   lockfile + 해시검증(`pip install --require-hashes`), SBOM 생성,
   `osv-scanner`/Dependabot로 알려진 취약점 감시, 게이트웨이 버전 핀.
   → DIP는 원래 "lock은 후속"이지만, **키를 쥐는 게이트웨이만은 예외로 lock+hash 우선 적용.**

### 3) 우리 아키텍처에 앉히는 법 (포트-어댑터)

`infrastructure/llm/` 에 **`LLMGateway` 포트**를 정의하고, 어댑터가 게이트웨이와 통신한다.
지금 [ADR-009]의 `Embedder` 포트, [ADR-006]의 Anthropic 어댑터와 **같은 패턴**.
모듈은 게이트웨이 구현이 아니라 인터페이스에 의존 → 나중에 교체도 어댑터만 손댄다.

## 근거

- **egress 제한이 왜 핵심인가**: 공급망 공격의 마지막 단계는 항상 "훔친 걸 밖으로 보내기"다.
  키 읽기(1차)는 완벽히 막기 어렵지만, 외부 전송(2차)은 네트워크에서 확실히 끊을 수 있다.
  약한 고리 하나를 확실히 끊는 게 여러 곳을 어설프게 지키는 것보다 낫다.
- **키 범위 최소화가 왜 중요한가**: 같은 사고가 나도 피해 범위가 provider 키로 한정된다(DB·Jira 안 샘).
  provider 키는 회사 콘솔에서 즉시 폐기·재발급 가능 = 회복 쉬움.
- **데이터 정책을 우리 코드에 남기는 이유**: 헌법 불변 규칙(Knowledge만 소비)을 서드파티 가용성·정책에
  묶으면 안 된다. 게이트웨이가 죽거나 바뀌어도 데이터 거버넌스는 그대로 살아있어야 한다.

## 절충 / 리스크

- **Self-host vs 관리형(SaaS) — 미결, 승인 필요**
  - Self-host(예: LiteLLM): 데이터가 밖으로 안 나가지만 **공급망 위험을 우리가 떠안음** → 위 4겹 격리 필수.
  - 관리형(예: Portkey/Cloudflare AI Gateway): 운영은 편하지만 **provider 키를 벤더에 맡기는** 신뢰 결정 → [APR-005] 성격의 승인 게이트 추가.
  - **잠정 추천**: 현 규모에선 **self-host + 4겹 격리**. 게이트웨이를 "똑똑한 정책 엔진"이 아니라 "격리된 키 보관·라우팅 프록시"로 **좁게** 써서 공급망 표면을 최소화.
- **새 의존성·인프라 증가**: 컨테이너 하나 + 네트워크 정책 운영 부담. 초기엔 provider 2곳만 라우팅으로 단순 유지.
- **egress 정책 운영**: provider 주소/CIDR가 바뀌면 갱신 필요. 관리형 secrets/network 정책과 함께 문서화.

## 결과 (승인 시 후속 작업 — 지금은 구현 안 함)

- `infrastructure/llm/`: `LLMGateway` 포트 + 어댑터. 기존 Anthropic 직접호출을 이 포트 뒤로.
- compose: 게이트웨이 별도 서비스 + **전용 시크릿 범위**(DIP 전체 `.env` 비공개) + egress 허용목록.
- 데이터 정책은 Context Builder에 유지(변경 없음) — 게이트웨이는 운영 거버넌스만.
- 의존성: 게이트웨이만 lock+hash 우선. `osv-scanner`/Dependabot 배선.
- **승인 게이트**: self-host/관리형 방향 결정 + (관리형 택할 시) 벤더 데이터 위탁 승인([APR-005] 계열).
