# ADR-006 — 지식 검색 아키텍처: 순수 벡터 RAG 대신 하이브리드

- 상태: Accepted (설계 방향)
- 날짜: 2026-07-07
- 관련: [Sprint-15](../tasks/Sprint-15.md) · [APR-004](../planning/approvals/APR-004-vector-store.md) · [../architecture/context-engine.md] · [../architecture/knowledge-lifecycle.md]

## 맥락
1:1 문의 처리 보조와 **레거시 "왜" 재구성**(퇴사한 기획자의 의도 복원)을 위해 과거 Jira/Git 지식을 검색·조립해야 한다.
"벡터DB + RAG로 관리하는 게 좋은가?"라는 질문에 대한 결정.

관찰:
- **순수 벡터 RAG는 유사도 검색엔 강하나 "왜"에 약하다.** "왜 이렇게 됐지"의 답은 유사도가 아니라
  **관계(기획 이슈→결정 코멘트→구현 커밋→이후 변경)와 시간축**이다. 벡터로 "비슷한 N개"를 뽑으면 이 사슬이 끊겨
  AI가 조각으로 그럴듯하게 지어낸다(퇴사자 지식 복원의 최대 리스크).
- **원본 청크를 그대로 임베딩하는 RAG는** 노이즈가 많고, 한국어 기술용어(에러코드/API명) 정밀도가 낮으며 환각이 잦다.

## 결정
순수 벡터 RAG 대신 **하이브리드 검색 아키텍처**를 채택한다:
1. **Postgres — 진실의 원천**: 이슈/코멘트/커밋/링크/상태/시간축. 정확 조회·필터(예: `project=ENG`→엔진팀)·조인.
2. **Knowledge 증류**: 이슈를 `{문제·원인·해결·배경}` + 출처로 LLM이 정제한 뒤 그것을 검색한다
   ("AI는 원천이 아니라 Knowledge를 소비" — 순수 RAG보다 신호↑ 환각↓).
3. **하이브리드 검색**: 키워드(정확·초기엔 Jira 자체 검색) + 벡터(의미) 융합.
4. **그래프**: 이슈↔커밋↔이슈 링크로 "왜" 사슬을 추적(우리가 만든 커밋 링크 활용).
5. **근거 인용 강제**: 모든 산출 문장에 이슈키/커밋 sha, 근거 없으면 "확실치 않음".

도입 순서(과설계 방지):
- **지금**: Jira 자체 검색 + LLM. 벡터 0.
- **스케일 시(키워드 재현율 부족)**: **pgvector**(기존 Postgres 내, 새 인프라 0). 별도 벡터DB 불필요.
- **임베딩 모델**: 사내 Jira(민감)라 **로컬 다국어 모델(bge-m3 등)** — 데이터 외부 유출 0. API 임베딩은 간단하나 데이터가 밖으로 나감.

## 근거
- "왜" 재구성은 관계/순서 문제 → 그래프+구조가 필수. 벡터 단독은 이를 못 준다.
- Knowledge 증류가 raw-chunk RAG 대비 정밀도·환각을 크게 개선.
- 키워드+벡터 하이브리드가 한국어 기술 텍스트에서 벡터 단독보다 정밀.
- pgvector 지연 도입으로 비용·운영 복잡도 최소화(YAGNI), 로컬 임베딩으로 프라이버시 확보.

## 절충 / 리스크
- 하이브리드는 순수 RAG보다 구현 요소가 많다 → 단계적 도입으로 완화(Jira검색부터).
- Knowledge 증류에 1회성 LLM 비용(백필) → 소액(수천 건 수천 원), [APR-005] 범위.
- 로컬 임베딩은 모델/컴퓨트 필요 → VPS에서 실행(배포 시 반영).

## 결과
- 검색은 `platform/context`(Context Builder) + `modules/search`/`modules/graph`로 조립, 원천은 Postgres.
- [APR-004](벡터스토어)는 **pgvector + 로컬 임베딩**으로 방향 확정(스케일 시 활성화).
- Sprint-15가 이 방향으로 MVP(Jira검색+LLM+근거)부터 구현.
